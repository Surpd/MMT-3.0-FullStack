# 2. Страница: Card Model (Данные и БД)

### **Суть:** Какие данные формируют сущность «Карточка фильма». Никакого UI, только чистая структура.

**1. Модель данных в коде (Data Structure)**
Объект, который мы собираем из базы и TMDB перед тем, как отдать во View.

Python

```python
class MovieCardModel(TypedDict):
    movie_id: int
    media_type: Literal["movie", "tv"]
    title: str
    year: int
    rating: float
    genres: list[str]
    poster_url: str | None
    
    # Специфичные данные
    runtime_mins: int | None
    directors: list[str]
    cast: list[str]  # Максимум 5 для оптимизации
    overview: str
    budget: str | None
    revenue: str | None
    
    # Данные сериалов
    tv_status: str | None
    next_episode_date: str | None
    
    # Пользовательский контекст
    user_status: Literal["liked", "watchlist", "archive", "none"]
```

**2. Хранение состояния пользователя (База Данных)**
Таблица `user_movies` в Supabase. Хранит только факты действий.

- `user_id` (bigint)
- `movie_id` (bigint)
- `media_type` (text)
- `status` (text) — строго `liked`, `watchlist`, `archive`.
- `rating` (smallint) — 1..5.
- `updated_at` (timestamptz).

### 2. Лимиты парсинга из TMDB (Анти-перегруз)

Чтобы не пробивать лимиты Telegram и не засорять память:

- **Актеры (`cast_spoiler`):** Парсим максимум 5 человек (3 для короткой версии, 5 для полной).
- **Студии (`studios`):** Оставляем максимум 2 главные студии.
- **Описание (`overview`):** Храним всё, но для короткой версии режем до ~400 символов.

# 4. Страница: User Actions

# **Суть:** Маппинг действий пользователя на конкретные функции (Handlers). Одно действие = одна функция.

**1. Навигация и переключение View**

- **Действие:** Клик `[ ℹ️ Детали ]` или `[ 🔙 Свернуть ]`
- **Payload:** `callback_data="view_toggle_{movie_id}_{is_full}"`
- **Функция:** `handle_view_toggle(callback: CallbackQuery)` -> вызывает рендер новой View.

**2. Изменение базовых статусов**

- **Действие:** Клик `[ ➕ Хочу ]`, `[ ✅ Смотрел ]`, `[ 🚫 Неинтересно ]`, `[ 🔄 Вернуть в базу ]`
- **Payload:** `callback_data="status_{new_status}_{movie_id}"`
- **Функция:** `handle_status_change(callback: CallbackQuery)` -> вызывает метод БД, обновляет View.

**3. Оценка (Рейтинг)**

- **Действие:** Клик по звездам `[ ⭐ 1-5 ]`
- **Payload:** `callback_data="rate_{movie_id}_{score}"`
- **Функция:** `handle_rating_change(callback: CallbackQuery)` -> обновляет поле `rating` в БД.

**4. Внешние действия**

- **Действие:** Клик `[ 🎬 Трейлер ]`
- **Payload:** `url="[https://youtube.com/](https://youtube.com/)..."`
- **Действие системы:** Открывает браузер (не требует хендлера).

---

Правило:
1 действие = 1 функция в коде


# 5. User Flows

## **Суть:** Сценарии (Journey) от первого клика до результата. Как взаимодействуют Model, View и Actions.

**Flow A: Базовый поиск и открытие фильма**

1. Пользователь отправляет текстовый запрос ("Интерстеллар").
2. Бот идет в TMDB API, возвращает список `MovieSearchResult`.
3. Пользователь кликает на фильм из списка инлайн-кнопок.
4. Слой данных собирает `MovieCardModel` (данные из TMDB + статус из БД).
5. Слой отображения генерирует `Short View` с клавиатурой `status=none`.
6. Бот присылает карточку.

**Flow B: Изменение статуса (Свайп / Оценка)**

1. Пользователь в `Short View` нажимает `[ ➕ Хочу ]`.
2. Срабатывает экшен `handle_status_change` с параметром `watchlist`.
3. Код делает UPSERT в БД `user_movies`, записывая статус `watchlist`.
4. Бот выводит Toast-уведомление (всплывашку) "⏳ Добавлено в список ожидания".
5. Код запрашивает новый рендер клавиатуры (где кнопка "Хочу" исчезнет, появится "Вернуть в базу").
6. Бот обновляет текущее сообщение (edit_caption) без отправки нового.

**Flow C: Запрос деталей**

1. Пользователь нажимает `[ ℹ️ Детали ]`.
2. Срабатывает экшен `handle_view_toggle`.
3. Код берет текущую `MovieCardModel`.
4. Слой отображения пропускает модель через шаблон `Full View` (без обрезки и спойлеров).
5. Клавиатура перерисовывается (кнопка меняется на `[ 🔙 Свернуть ]`).
6. Бот обновляет текущее сообщение (edit_caption).
